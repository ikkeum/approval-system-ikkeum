# Design: 사내 전자결재 시스템 (Ikkeum Approval)

## Problem Statement

10명 이하 소규모 팀이 현재 **카카오톡으로 연차·품의를 결재** 중.
문제는 속도가 아니라 **기록·검색 불가능**: 3개월 뒤 "이거 승인받았나요?" 하면 아무도 모름.
Vercel + Supabase **무료 티어**로 사내 전용 결재 시스템 구축.

## Demand Evidence

- 실사용자 ≤10명, 현재 카톡 결재 (기록 누락이 실제 pain)
- 사내 전용이라 외부 demand 검증은 N/A. 대신 **사내 도입 성공 지표**로 갈음:
  - 배포 후 4주간 연차·품의 100%가 시스템으로 올라와야 함 (카톡 결재 0건)
  - 3개월 뒤 "그때 그 품의" 검색이 10초 안에 되어야 함

## Status Quo

- **도구**: KakaoTalk (메신저)
- **흐름**: 신청자 → 카톡 DM/그룹방 → 팀장 "OK" → 끝
- **비용**: 무형 비용 높음 — 기록 없음, 휴가일수 집계 수동, 품의 누적 이력 없음

## Target User & Narrowest Wedge

- **Target**: 사내 모든 임직원 (~10명), 팀장 1명
- **Narrowest Wedge**: 연차 + 품의 동시 (목업 범위 그대로). 1단계 결재.
  - 이유: 둘 다 스키마·UI 구조 거의 같아서 따로 나누는 게 오히려 낭비

## Constraints

- **비용 0원** (Vercel Hobby + Supabase Free)
- **사내 전용** (도메인 화이트리스트로 외부 차단)
- **알림 필수** (Google Chat Webhook — GWS 사용 중)
- **MVP 배포 기한**: 1주

## Premises (합의됨)

1. ✅ 무료 티어 용량 문제 없음 (10명 × 연 1000건 = <10MB)
2. ✅ 카톡의 진짜 문제는 기록/검색. 알림 없으면 시스템 사장.
3. ✅ 권한 경계는 **Supabase RLS**로 (앱 코드 if문 아님)
4. ✅ 도메인 제한은 **Google SSO의 `hd` 파라미터**로 (OAuth 레벨에서 차단)
5. ✅ 파일 첨부는 Supabase Storage (1GB 무료)

## Recommended Approach: A — 미니멀 바닐라

### Stack

- **Frontend**: Next.js 15 App Router (Vercel Hobby)
- **Auth**: Supabase Auth 이메일/비번 + 도메인 화이트리스트 (v1). GWS Admin 확보 후 Google SSO 전환.
- **DB**: Supabase Postgres (RLS 필수)
- **Storage**: Supabase Storage (영수증·첨부)
- **Notification**: Google Chat Incoming Webhook (Supabase Edge Function 호출)
- **직인(개인 도장)**: [sign-generator](https://github.com/ketaro01/sign-generator) + 사내 배치 폰트(MISEANG 등). 가입 시 1회 생성해 `profiles.stamp_svg`에 저장, 승인/반려 시점에 해당 결재자의 SVG를 렌더링.

### Schema (eng-review 반영)

```sql
-- 사용자 (Supabase auth.users 연결)
create table profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  email text unique not null,
  name text not null,
  dept text,
  role text check (role in ('member','manager','admin')) default 'member',
  manager_id uuid references profiles(id) on delete set null,
  created_at timestamptz default now()
);

-- 결재 문서 (연차·품의 통합). payload는 jsonb, 앱 레벨에서 Zod로 검증.
create table approvals (
  id bigserial primary key,
  type text check (type in ('leave','expense')) not null,
  title text not null,
  author_id uuid not null references profiles(id),              -- RESTRICT (이력 보존)
  approver_id uuid references profiles(id),                     -- DRAFT일 때 null 허용
  status text check (status in ('DRAFT','PENDING','APPROVED','REJECTED','CANCELED')) default 'DRAFT',
  payload jsonb not null,
  attachments jsonb not null default '[]'::jsonb,               -- [{path, name, size}]
  created_at timestamptz default now(),
  submitted_at timestamptz,
  decided_at timestamptz,
  decision_comment text,
  updated_at timestamptz default now(),
  check (status = 'DRAFT' or approver_id is not null)
);

create index idx_approvals_approver_status on approvals(approver_id, status);
create index idx_approvals_author_status on approvals(author_id, status);

-- updated_at 자동 갱신
create or replace function set_updated_at() returns trigger language plpgsql as $$
begin new.updated_at = now(); return new; end $$;
create trigger trg_approvals_updated
  before update on approvals
  for each row execute function set_updated_at();

-- 액션 히스토리 (감사 추적)
create table approval_actions (
  id bigserial primary key,
  approval_id bigint not null references approvals(id) on delete cascade,
  actor_id uuid not null references profiles(id),
  action text not null check (action in ('submit','approve','reject','cancel','comment')),
  comment text,
  created_at timestamptz default now()
);

create index idx_actions_approval on approval_actions(approval_id);

-- profile 자동 생성 (최초 SSO 로그인 시 auth.users → profiles)
create or replace function public.handle_new_user() returns trigger language plpgsql security definer as $$
begin
  insert into public.profiles (id, email, name)
  values (new.id, new.email, coalesce(new.raw_user_meta_data->>'name', split_part(new.email,'@',1)));
  return new;
end $$;
create trigger on_auth_user_created
  after insert on auth.users
  for each row execute function public.handle_new_user();

-- 상태 전이 강제 (approver가 status를 이상하게 바꾸지 못하도록)
create or replace function enforce_status_transition() returns trigger language plpgsql as $$
begin
  if old.status = 'DRAFT'    and new.status not in ('DRAFT','PENDING') then raise exception 'bad transition'; end if;
  if old.status = 'PENDING'  and new.status not in ('APPROVED','REJECTED','CANCELED') then raise exception 'bad transition'; end if;
  if old.status in ('APPROVED','REJECTED','CANCELED') and new.status <> old.status then raise exception 'terminal'; end if;
  return new;
end $$;
create trigger trg_status_transition
  before update of status on approvals
  for each row execute function enforce_status_transition();
```

### RLS (반드시 ENABLE 먼저)

```sql
alter table profiles          enable row level security;
alter table approvals         enable row level security;
alter table approval_actions  enable row level security;

-- profiles: 인증된 사용자는 조회 가능 (UI에서 이름 표시용). 본인만 수정.
create policy "profiles_read" on profiles for select to authenticated using (true);
create policy "profiles_self_update" on profiles for update to authenticated
  using (id = auth.uid()) with check (id = auth.uid());

-- approvals SELECT: 본인 문서 OR (본인이 결재자 AND DRAFT 아님)
create policy "approvals_select" on approvals for select to authenticated
  using (
    author_id = auth.uid()
    or (approver_id = auth.uid() and status <> 'DRAFT')
  );

-- INSERT: 본인으로만 생성
create policy "approvals_insert" on approvals for insert to authenticated
  with check (author_id = auth.uid());

-- Author 수정: DRAFT 편집 + PENDING을 CANCELED로 철회
create policy "approvals_author_update" on approvals for update to authenticated
  using (
    author_id = auth.uid()
    and status in ('DRAFT','PENDING')
  )
  with check (author_id = auth.uid());

-- Approver 수정: PENDING을 APPROVED/REJECTED로만 (WITH CHECK으로 탈취 방지)
create policy "approvals_approver_decide" on approvals for update to authenticated
  using (approver_id = auth.uid() and status = 'PENDING')
  with check (approver_id = auth.uid());

-- DELETE: DRAFT만 본인이
create policy "approvals_delete_draft" on approvals for delete to authenticated
  using (author_id = auth.uid() and status = 'DRAFT');

-- approval_actions SELECT: 연결된 approval이 보이는 사람만
create policy "actions_select" on approval_actions for select to authenticated
  using (exists (
    select 1 from approvals a
    where a.id = approval_id
      and (a.author_id = auth.uid() or (a.approver_id = auth.uid() and a.status <> 'DRAFT'))
  ));

-- actions INSERT: actor=self + 해당 approval의 author/approver만
create policy "actions_insert" on approval_actions for insert to authenticated
  with check (
    actor_id = auth.uid()
    and exists (
      select 1 from approvals a
      where a.id = approval_id
        and (a.author_id = auth.uid() or a.approver_id = auth.uid())
    )
  );
```

### Storage RLS (첨부파일 버킷 `approval-attachments`)

```sql
-- 경로 규칙: {user_id}/{approval_id}/{filename}
-- 업로드: 본인 prefix만
create policy "storage_upload_own" on storage.objects for insert to authenticated
  with check (bucket_id = 'approval-attachments' and (storage.foldername(name))[1] = auth.uid()::text);

-- 다운로드: 업로더 본인 + approval의 approver
create policy "storage_read" on storage.objects for select to authenticated
  using (
    bucket_id = 'approval-attachments'
    and (
      (storage.foldername(name))[1] = auth.uid()::text
      or exists (
        select 1 from approvals a
        where a.id::text = (storage.foldername(name))[2]
          and a.approver_id = auth.uid()
          and a.status <> 'DRAFT'
      )
    )
  );
```

**핵심 원칙 3가지**

1. **RLS ENABLE은 테이블 생성 직후** — 데이터 들어간 뒤 켜면 이미 유출
2. **handle_new_user trigger는 첫 로그인 전** — 없으면 profiles 빈 상태에서 RLS join 실패 → 모든 쿼리 빈 결과 (디버깅 지옥)
3. **상태 전이는 DB trigger로** — 앱 if문 믿지 말 것. RLS + trigger 이중 방어

## What I noticed about how you think

- 이미 목업을 30KB짜리 jsx로 구체화해둔 점 — 생각만 많은 사람이 아니라 **손을 움직이는 사람**. 결재선 질문에 바로 "1단계면 충분"이라고 답한 것도 불필요한 복잡성을 본능적으로 피한다는 신호.
- 카톡을 "현재 쓰는 도구"로 솔직하게 답함 — 많은 사람이 여기서 "우리는 원래 체계가 있는데 개선하려고..." 같은 합리화를 함. 있는 그대로 보는 태도.
- 무료 티어를 제약이 아니라 설계 원칙으로 받아들인 점 — 10명한테 쓰는 툴에 월 $20짜리 Pro 결제는 과함. 이 판단을 처음부터 한 게 정확.
